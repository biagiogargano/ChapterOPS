# Questionnaire / Report Tasks — Planning Spec

**Status: PLANNING ONLY.** No code, schema, RLS, RPC, auth, flags, or data-model
changes in this document or this checkpoint. This is the design that a later,
separately-approved build phase will implement. Lives on
`feature/questionnaire-reports-planning`; **not** to be merged into `phase-2`
without approval. Expands `PRODUCT_BACKLOG.md` §6.

---

## 1. What this is

A new **task kind** where "completing" the task means **submitting structured
answers to a questionnaire** — not a proof upload and not a one-tap done. It
introduces:
- a **questionnaire definition** (the questions),
- an **answer/response model** (what a responder submitted),
- a **draft → submit workflow** distinct from the existing submit→approve flow.

It sits **on top of** the existing deterministic template/task/recurrence engine.
The engine still generates the tasks; this just adds a new completion mechanism.

## 2. Anchor use case — weekly officer report

- A recurring weekly report task, generated by the existing recurrence engine.
- An officer can **edit answers throughout the week** (the window may span the
  weekend).
- A **final submit** confirms/locks the answers (snapshot).
- Example fields: goal/status update · target value vs current value · a
  **"No update"** shortcut · "do you need help with anything?" · announcements ·
  "what else are you working on?"

Other use cases (same machinery): polls, surveys, RSVP questionnaires, goal
updates, meeting prep, and (further out) class/assignment/quiz-style flows.

## 3. Question / response types (eventual full set)

Build in tiers so an MVP can ship before the long tail:

**Tier 1 (MVP — covers the weekly officer report):**
- short text · long text · number · single-select (multiple choice) ·
  multi-select (checkbox) · a **"No update"** shortcut

**Tier 2:**
- percentage · **current value vs target value** (paired numeric) · date picker ·
  time picker

**Tier 3:**
- **generated time-slot picker** (from start/end + interval — reusable for
  availability) · poll/vote · file/proof attachment

## 4. Workflow / state model (the hard part)

The existing task state machine is **submit → approve/reject** (a reviewer gate).
Questionnaire tasks need a **different** lifecycle:

```
not_started → in_progress (answers being drafted/edited, autosaved)
            → submitted   (final submit; answers snapshotted/locked)
            → [optional] reopened (officer/admin reopens before deadline)
```

Open design questions (to resolve in the design phase, NOT now):
- **Edit-after-submit:** locked on submit, or editable until the window closes?
  (Anchor use case implies editable until deadline, with an explicit final submit.)
- **Does it need approval at all?** The officer report is likely **self-submit, no
  reviewer**. Keep questionnaire submission *decoupled* from the approve gate so we
  don't entangle the two state machines.
- **Window/deadline:** open + close timestamps, driven by the recurrence cadence.
- **"No update" shortcut:** does it count as a valid submission for the week?

> Decision to preserve: **do not modify the existing submit→approve state machine.**
> Questionnaire tasks get their own parallel lifecycle so the current alpha flows
> are untouched.

## 5. Data model (DESIGN ONLY — do not create yet)

Sketch for the future schema phase. **Not** to be built in planning.

- **questionnaire definition** — ordered list of questions, each with: id, type
  (see §3), prompt, required flag, options (for select types), config (min/max,
  interval for time-slot, target for value-vs-target). Likely attached to a
  template or a task kind.
- **response** — keyed to **(task instance + responder)**; holds per-question
  answers + status (in_progress/submitted) + submitted_at. Today tasks only store
  proof text/state, so this is a genuine new table/shape.
- **answer** — per-question value; shape varies by type (text, number, selected
  option id(s), date/time, slot, file ref).

Recurrence reuse: each weekly occurrence is its own task instance → its own
response row, exactly like prep tasks generate per occurrence today.

## 6. Reusable UI (build on existing patterns)

- A **question renderer** (one component per question type, dispatched by type —
  mirrors the `QuickActionCard` dispatcher pattern already in Today).
- The **generated time-slot picker** (Tier 3) — can build on the existing
  `SearchablePicker` / form patterns.
- A **draft/submit form screen** with autosave-on-edit and an explicit final
  submit button.

## 8. Suggested phasing (each its own approved checkpoint)

1. **Spec finalization** (this doc → resolve §4 open questions; lock Tier-1 types).
2. **Answer data model + draft/submit workflow** design (schema phase — separate
   approval; this is where schema/RLS/RPC finally change).
3. **Question renderer + draft/submit UI** (Tier-1 types).
4. **Recurrence/window wiring** (weekly officer report end-to-end).
5. **Tier-2 / Tier-3 question types** incrementally.
6. **#8 meeting-agenda** consumes the answers.

## 9. Explicit non-goals for now

- No AI (drafting from natural language stays direction-only).
- No schema/RLS/RPC/auth/flags/data-model changes in the planning checkpoint.
- No change to the existing submit→approve task state machine.
- No server-shared templates work.
- Nothing merges to `phase-2` without approval.
